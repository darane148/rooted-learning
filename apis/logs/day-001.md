### What Happened

**BOLA Test (Broken Object Level Authorization):**
```
$ curl -H "Authorization: Bearer $TOKEN" http://localhost:8888/identity/api/v2/user/2
{
  "id": 2,
  "email": "victim@test.com",
  "name": "Victim User",
  "number": "9876543210",
  "role": "user"
}
```

I was logged in as user 1 (attacker) but successfully read user 2's profile by changing the ID in the URL. The API never checked if the authenticated user owns resource ID 2. This is BOLA — one of the most common and critical API vulnerabilities.

**JWT Decode:**
```
Header: {"alg":"HS256"}
Payload: {"sub":"attacker@test.com","iat":1680000000,"exp":1680086400}
```

The payload contains `sub` (subject = email), `iat` (issued at), `exp` (expiration). Changing `sub` to another user's email and resigning with the secret would impersonate them.

---

### Mistakes Made

1. **Forgot to add `=` padding when base64 decoding JWT** — got a decode error. JWT base64 is URL-safe (uses `-` and `_` instead of `+` and `/`) and strips padding. Add `==` when decoding manually.
2. **Sent Authorization header as `JWT` instead of `Bearer`** — got 401. The API expected `Bearer <token>`, not `JWT <token>`. Always check the API docs or intercept a working request in Burp first.
3. **Tried to modify the JWT payload manually without re-signing** — the server rejected it (signature verification failed). Without the secret, you can't forge a valid HMAC-signed JWT.

---

### What I Learned

- **JWT structure**: `header.payload.signature` — all base64url-encoded. The signature verifies integrity, but if the secret is weak or the `none` algorithm is accepted, it can be bypassed.
- **BOLA** (Broken Object Level Authorization) = IDOR at the API level. The API accepts valid auth but doesn't check if the user *owns* the requested resource.
- **`none` algorithm attack**: some old JWT implementations accept `alg: none`, meaning no signature required. `jwt_tool -X a` tests for this.
- API documentation endpoints (`/swagger-ui.html`, `/api-docs`, `/openapi.json`) are often left exposed and reveal the entire attack surface.
- Always check API responses for unexpected data — a 200 response for another user's data is a vulnerability even if there's no error.
